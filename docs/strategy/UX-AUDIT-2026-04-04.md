# Glass UX Audit — 2026-04-04

Consolidated feedback from Justin's critical review of the Flint Glass interface.
This document is the single source of truth for all UX issues identified in this session.

Issues are organized by **cross-cutting theme** rather than screen area, because many problems share the same root cause and should be fixed together.

---

## Theme 1: Trust Gap — "Does this tool actually work?"

The most damaging pattern. Multiple features look functional but do nothing when interacted with. A designer who discovers 2-3 dead interactions will stop trying new ones.

**Priority: P0 — fix before any new features.**

### T1.1 Governance rule toggles don't affect violations
- **What:** Toggling a rule off in the Governance Rules panel updates Zustand state but violations still appear in the dashboard. The linter runs independently and ignores overrides.
- **Where:** [GovernanceDashboard.tsx:694](src/components/ui/GovernanceDashboard.tsx#L694) — `effectiveLinterWarnings` and `effectiveA11yWarnings` are computed from raw linter output without consulting `governanceStore.overrides`.
- **Root cause:** Two-layer disconnect — (1) overrides live in governanceStore but the dashboard reads raw violations from canvasStore/editorStore, (2) `saveRuleOverrides`/`getRuleOverrides` IPC is not wired in preload.ts or main.ts, so persistence falls back to localStorage which the linter never reads.
- **Fix direction:** Filter `effectiveLinterWarnings` and `effectiveA11yWarnings` against `governanceStore.overrides` — skip any violation whose rule has `enabled: false`. Wire the IPC persistence channel.

### T1.2 Toggle-back still shows "Modified"
- **What:** Toggling a rule off then back on still shows the "Modified" badge. The toggle should undo cleanly.
- **Where:** [GovernancePanel.tsx:357-370](src/components/ui/GovernancePanel.tsx#L357-L370) — `handleToggle` always calls `setOverride`, never `resetOverride` when the new value matches the default.
- **Fix direction:** In `handleToggle`, check if the new `enabled` value matches the rule's default. If so, call `resetOverride(ruleId)` instead of `setOverride`.

### T1.3 "Pending style changes — review" does nothing on click
- **What:** The export gate label in the StatusBar says "Pending style changes — review before export" but clicking it provides no feedback. A designer doesn't know what this means or what to do.
- **Where:** [StatusBar.tsx:139](src/components/editor/StatusBar.tsx#L139)
- **Fix direction:** Make it clickable — navigate to the Governance tab on click. Add a tooltip explaining what "pending style changes" means. Consider renaming to something more actionable: "3 issues blocking export — view".

### T1.4 Sidebar doesn't resize
- **What:** The ResizeHandle component and wiring exist ([App.tsx:1052](src/App.tsx#L1052)) but dragging the handle does nothing visible.
- **Where:** Likely a z-index issue — React Flow's event layer captures pointer events over the 4px-wide handle.
- **Fix direction:** Increase the handle's z-index or expand its hit area. The handle has a wider invisible hit area (`-inset-x-1.5`) but the React Flow overlay may be eating events.

### T1.5 Undo after demo load is broken
- **What:** Clicking the Demo button loads a component, then pressing undo says "undone" but the preview doesn't revert. The undo stack doesn't have a previous state to restore to.
- **Where:** Demo load uses `syncCode()` which replaces editor state without pushing to the undo stack.
- **Fix direction:** Either push the previous state onto the undo stack before `syncCode()`, or suppress the "undone" notification when there's nothing to undo.

## Theme 2: Information Architecture — "Where does this belong?"

Glass has no clear hierarchy of importance. Features are buried or misplaced. Important actions are hidden where users won't find them, while prominent real estate is wasted on labels and status text.

**Priority: P1-P2 — address as a cohesive layout rework.**

### T2.1 Header underutilized, footer overloaded
- **What:** The header has the app title, file name (tiny), a search button, and settings gear. The footer (StatusBar) has: export gate, violation count, save state, sync status, Figma status, responsive breakpoint, MCP connection, Autopilot, and "Open your project" link. The footer is doing too much.
- **Where:** Header: [App.tsx:838-943](src/App.tsx#L838-L943). Footer: [StatusBar.tsx](src/components/editor/StatusBar.tsx)
- **Fix direction:** Header left: app title + file name (larger). Header center/left: issue count badge (links to Governance tab). Header right: search (more prominent), settings. Footer: connection status, save state, Figma sync — pure telemetry.

### T2.2 File name, Issues, and "Open your project" belong in the header
- **What:** These are primary navigation/status elements hidden in the footer where they're hard to see.
- **Fix direction:** Relocate to the header bar where they get proper visual weight.

### T2.3 Search/Command Palette is undersold
- **What:** The Command Palette trigger (`Cmd+K`) is a small icon button pushed to the far right. It's the most powerful navigation tool in Glass but looks like a minor utility.
- **Where:** [App.tsx:870-880](src/App.tsx#L870-L880)
- **Fix direction:** Make it a wider search bar (like Raycast/Spotlight) floated left or center. Show placeholder text like "Search components, actions, files... ⌘K".

### T2.4 Run Audit is stuck in a cramped sidebar header
- **What:** Run Audit sits in the sidebar header alongside title, delta badge, and override count. The header is too narrow for all this. Run Audit is an action, not navigation.
- **Where:** [GovernanceDashboard.tsx:1467-1496](src/components/ui/GovernanceDashboard.tsx#L1467-L1496)
- **Fix direction:** Move to: (a) Command Palette (already exists), (b) inside the export gate banner as a secondary action, or (c) a floating action in the issue list area. The sidebar header should be for navigation only.

### T2.5 Settings buried as hidden content
- **What:** Advanced Insights, Developer Controls, Autopilot toggle, Top Triggered Rules, Session & Baseline, Agent Activity, Audit Log, Compliance Coverage, and Config Inheritance are all collapsed at the bottom of the Governance tab. This mixes settings, diagnostics, and configuration.
- **Fix direction:** Separate concerns: (a) Settings/toggles belong in a Settings modal or Command Palette. (b) Diagnostics (Agent Activity, Audit Log) could be a separate tab or live in the MCP event stream. (c) Config display is developer-facing and could live behind a "Developer" toggle.

### T2.6 Autopilot toggle doesn't belong in Advanced Insights
- **What:** The Autopilot on/off toggle is nested inside "Advanced Insights" which a designer would never find. If Autopilot is valuable, it should be prominently placed.
- **Fix direction:** Move to the status bar (where it was before) or make it a top-level toggle in the Governance tab header. The toggle should be visible when violations exist.

### T2.7 Annotations buried in Properties panel
- **What:** The annotation/comment system is hidden at the bottom of the Properties panel in the right sidebar. Comments and annotations are a collaboration-level feature — they should be as prominent as search or settings.
- **Where:** AnnotationList renders inside the Properties tab. Annotations are also shown as dots on the LayerTree.
- **Fix direction:** Add an annotations icon/badge to the header bar (next to search and settings). Clicking it opens a dedicated annotation panel or popover. When annotations exist, show a count badge.

## Theme 3: Contrast & Readability — "I can't read this"

Ironic for a governance tool that enforces WCAG contrast compliance. These undermine credibility.

**Priority: P1 — quick wins that fix credibility.**

### T3.1 Gray text fails WCAG across the dashboard
- **What:** `text-zinc-600` on dark backgrounds has ~2.5:1 contrast ratio, failing WCAG AA (needs 4.5:1). Used extensively in GovernanceDashboard for secondary text like "How is this calculated?", descriptions, and hints.
- **Where:** Multiple locations in GovernanceDashboard.tsx — search for `text-zinc-600`. Also in GovernancePanel.tsx rule descriptions.
- **Fix direction:** Replace `text-zinc-600` with `text-zinc-400` or `text-zinc-500` minimum across the entire dashboard. Audit all secondary text for contrast compliance.

### T3.2 Dark canvas background — text unreadable
- **What:** Preview node wrapper is too dark (`bg-gray-950`). Some text inside the preview blends into the surrounding chrome, requiring guesswork.
- **Where:** [XYCanvas.tsx:137](src/components/editor/XYCanvas.tsx#L137)
- **Fix direction:** Lighten the node wrapper or add a distinct border/background boundary. Consider `bg-gray-900` or a subtle inner border.

### T3.3 White scrollbar background
- **What:** The scrollbar in the left sidebar has a white/light background that is visually jarring against the dark theme.
- **Fix direction:** Add custom scrollbar styles via CSS — `scrollbar-color` and `::-webkit-scrollbar` rules targeting the sidebar containers. Match the scrollbar track to the panel background.

## Theme 4: Information Overload — "Too much before the thing I need"

Every element competes for attention. When everything is visible, nothing is. Glass needs progressive disclosure — show the verdict and the issues, let users dig deeper on demand.

**Priority: P1 — compact the health section, reduce noise.**

### T4.1 Governance health section is cluttered
- **What:** From top to bottom, the user sees 15+ distinct UI elements before reaching the actual issues: header, run audit, export gate banner, effort framing, delta banner, score ring, grade letter, sparkline, score trend hint, next step text, rewind link, sub-score breakdown (3 rows), "how is this calculated?", zone divider, category chips. The health section consumes ~40% of visible viewport.
- **Where:** [GovernanceDashboard.tsx:1463-1660](src/components/ui/GovernanceDashboard.tsx#L1463-L1660)
- **Fix direction:** Compact drastically. Proposed layout: export gate banner (1 line) → score ring + grade + one coaching sentence (compact row) → issue list. Move sub-scores, sparkline, and trend hints into the "Advanced insights" collapsible.

### T4.2 "3 issues" repeated 4 times
- **What:** The same violation count appears in: (a) export gate banner, (b) effort framing text, (c) score trend hint, (d) next step coaching text. The section over-explains what the score ring already shows.
- **Where:** [GovernanceDashboard.tsx:1516-1650](src/components/ui/GovernanceDashboard.tsx#L1516-L1650)
- **Fix direction:** The export gate banner is the primary signal. The score ring + grade is the emotional anchor. Everything between those and the issue list is commentary. Collapse to: banner → compact score → issues. One sentence max for coaching.

### T4.3 Drag handle chrome — no user value
- **What:** The "Drag to reposition" bar is always visible but provides no value with a single preview node. The window frame also makes the ViolationIndicator (red dot) look like a window close button.
- **Where:** [XYCanvas.tsx:145-155](src/components/editor/XYCanvas.tsx#L145-L155)
- **Fix direction:** Options: (a) hide the chrome bar entirely, (b) show only on hover with reduced prominence, (c) remove the text and keep only a minimal drag affordance. The red dot should not look like a window close button — consider a different shape or position.

### T4.4 Code preview auto-pops on every node selection
- **What:** The GhostCodeSnippet appears automatically every time a node is selected. For designers, this is noise. For developers reviewing what the AI generated, it's useful — but only sometimes.
- **Where:** [GhostCodeSnippet.tsx](src/components/editor/GhostCodeSnippet.tsx)
- **Fix direction:** Change to opt-in: add a "View Source" button in the Properties panel or a toolbar button on the canvas node. Keep the `Esc` dismiss, but don't auto-show. Alternatively, make it a user preference that defaults to off.

### T4.5 Remove Quick Load / Load Demo buttons
- **What:** "QUICK LOAD" label and "Load Demo" button are dev-only (`import.meta.env.DEV`) but still visible during development. They serve no purpose now that the Launch Screen handles demo flow.
- **Where:** [LivePreview.tsx:1153-1168](src/components/editor/LivePreview.tsx#L1153-L1168)
- **Fix direction:** Remove the Quick Load / Load Demo section entirely.

## Theme 5: Missing Interactions — "I expected this to work"

Features that exist but lack the connections to be useful. These add real value but only matter once the foundation is solid.

**Priority: P2-P3 — build after trust and readability are fixed.**

### T5.1 Hovering a violation doesn't highlight on canvas
- **What:** Expanding a violation card selects the node in Zustand (`setSelectedNode`) but there's no visual highlight in the iframe preview. The user can't see which element the issue refers to.
- **Where:** [GovernanceDashboard.tsx:610-612](src/components/ui/GovernanceDashboard.tsx#L610-L612) — only updates store, no postMessage to iframe.
- **Fix direction:** On violation hover/expand, send `{ type: 'HIGHLIGHT_NODE', nodeId }` via postMessage to the preview iframe. Add corresponding CSS highlight logic (pulsing outline, scroll-into-view) in the iframe's injected script.

### T5.2 Preview window is not resizable
- **What:** Preview is hardcoded to 900x600px. Users cannot make it taller, wider, or smaller.
- **Where:** [XYCanvas.tsx:138](src/components/editor/XYCanvas.tsx#L138) — `style={{ width: 900, height: 600 }}`
- **Fix direction:** Add `NodeResizer` from `@xyflow/react` to the LivePreviewNode. Allow free-form resize with reasonable min/max constraints (e.g. min 400x300). Persist the last-used size in canvasStore.

### T5.3 Code preview not horizontally resizable
- **What:** The code block container is fixed at 480px wide. Long lines get cut off. Vertical resize works but horizontal does not.
- **Where:** [GhostCodeSnippet.tsx:279](src/components/editor/GhostCodeSnippet.tsx#L279) — `w-[480px]` fixed width.
- **Fix direction:** Either make the container horizontally resizable, widen the default (600-700px), or allow the snippet to dock to the bottom of the canvas as a full-width panel.

### T5.4 Code preview is read-only — no constrained actions
- **What:** The floating code snippet shows ~9 lines of syntax-highlighted source when a node is selected. It's not actionable.
- **Where:** [GhostCodeSnippet.tsx](src/components/editor/GhostCodeSnippet.tsx)
- **Recommendation:** Don't add freeform code editing (breaks Glass's observability-only identity and Commandment 13). Instead, make the snippet actionable through constrained interactions:
  - Click a class name to see its token mapping and swap it from a dropdown
  - Click a text string to inline-edit just that string
  - Right-click for "Copy to clipboard" / "Open in IDE" / "Fix this line"

### T5.5 Assets tab — point to a library
- **What:** The Assets tab was removed but Justin wants it back as a key differentiator: point to a UI component library, assets populate in Glass, designers drag-and-drop into work. The current ComponentPanel has drag-to-insert but lacks the "point to a library" flow.
- **Where:** [ComponentPanel.tsx](src/components/ui/ComponentPanel.tsx)
- **Fix direction:** Future sprint — wire up `flint_set_library` + `flint_add_remote_library` MCP tools to populate the component panel from real npm packages. Make the panel feel like a design tool asset browser.

### T5.6 Governance Panel tab width shift (FIXED)
- **What:** Changing tabs in the Governance Rules panel caused the dialog width to shift because content-dependent sizing had no minimum constraint.
- **Where:** [GovernancePanel.tsx:445](src/components/ui/GovernancePanel.tsx#L445)
- **Status:** Fixed — dialog now uses `w-[48rem] max-w-[calc(100vw-2rem)]` for stable width.

## Execution Order

Based on the themes above, this is the recommended burn-down sequence:

| Phase | Theme | Items | Rationale |
|-------|-------|-------|-----------|
| **1** | Trust Gap | T1.1–T1.5 | Fix every dead interaction before adding anything new |
| **2** | Contrast | T3.1–T3.3 | Quick wins that fix credibility |
| **3** | Overload | T4.1–T4.5 | Compact health section, reduce noise |
| **4** | IA Rework | T2.1–T2.7 | Restructure header/footer, surface buried features |
| **5** | Interactions | T5.1–T5.5 | Canvas highlight, resize, constrained code actions |

Within each phase, items are ordered by implementation complexity (simplest first) to maximize momentum.
