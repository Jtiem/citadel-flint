# FIXTURE.1.1 — Security Review

**Phase:** FIXTURE.1.1 (DTCG Token Shape Adapter)
**Dimension:** security
**Reviewer:** flint-security-reviewer
**Date:** 2026-04-19
**Round:** 1
**Verdict (derived):** BLOCK

## Summary

The adapter is a pure in-process transform with no `fs`, `child_process`, `path`, or network imports (Commandments 13/14 pass). ID-collision safety with DB-issued positive ids is sound (signed-int32 forced-negative FNV-1a). Regex is not used on source code. Hash-space is disjoint.

However, the adapter fails the "never throws and never stack-overflows" guarantee from contract §2 under adversarial input. Three blocking issues were reproduced by tracing adversarial payloads through the code paths:

1. **Unbounded recursion in `_walkDTCG`** — no depth guard
2. **`String(Symbol)` throws** — violates the "adapter never throws" invariant
3. **Non-string `$ref` crashes `_lookupByDotPath`** — `split` on non-string throws

Plus two warnings (prototype-chain read via bracket access in `_lookupByDotPath`; O(N²) visited-set via `Array.includes`) and one suggestion (undocumented hash-collision probability).

Because `deriveVerdict(findings, 'security')` escalates any blocking security finding to **BLOCK**, this phase cannot ship until the three blockers are remediated. All fixes are one-line or single-function scope.

## Findings

### BLK-1 — Unbounded recursion in `_walkDTCG` (DoS)

- **File:** `flint-mcp/src/core/dtcgTokenAdapter.ts:170-229`
- **Commandment:** 13 (Deterministic Surgery must not destabilize the engine)
- **Observed:** `_walkDTCG` recurses into every nested group via `_walkDTCG(root, val as DTCGDocument, childPath, tokens, errors)` with no depth guard. `isDTCGDocument` has a `depth > 20` guard (line 119) but that guard is **only on shape detection, not on flatten**.
- **Rationale:** A malicious `design-tokens.json` (or a honest-but-buggy one generated by an Envoy sync glitch) with ~10,000 levels of nesting causes a V8 RangeError in the MCP engine. Because this runs inside `server.ts:2038` during the audit path, the entire `audit_ui_component` request dies with an unhandled exception. An attacker who can place a file at `design-tokens.json` in any workspace the MCP server audits (a Figma ingestion result, a GPX pack import) has a one-byte DoS.
- **Proposed fix:** Add `depth` parameter with a cap (e.g., 64 — deeper than any real design system) to `_walkDTCG`; on exceed, push a `TokenAdapterError` with a new code `DEPTH_EXCEEDED` and return early.
- **Scope:** one-file
- **Status:** open

### BLK-2 — `String(rawValue)` throws on Symbol / throwing getter

- **File:** `flint-mcp/src/core/dtcgTokenAdapter.ts:211`
- **Commandment:** 13
- **Observed:** The fallback branch `resolvedValue = String(rawValue)` assumes `rawValue` is stringifiable. `String(Symbol())` throws `TypeError: Cannot convert a Symbol value to a string`. A `$value` that is an object with a throwing `toString`/`Symbol.toPrimitive` also throws here.
- **Rationale:** Contract §2 asserts "the adapter never throws". Any throw here propagates out of `normalizeTokenShape` → `server.ts:2038` → the `audit_ui_component` tool handler, which does not wrap this path in try/catch per the contract's "no changes to any visitor" scope. While Symbols cannot arrive via `JSON.parse`, the adapter's public surface is `normalizeTokenShape(raw: unknown)` and the contract type is `TokenAdapterResult`, not "only-JSON-safe input". Programmatic callers (tests, swarm.ts, future library adapters) can pass such values, and the contract does not forbid it.
- **Proposed fix:** Wrap `String(rawValue)` in try/catch, and treat failure as a broken value: push a `TokenAdapterError` with a new `INVALID_VALUE_TYPE` code (or reuse `ALIAS_BROKEN_REF` with a note) and `continue`. Equivalent guard: check `typeof rawValue !== 'symbol' && typeof rawValue !== 'function'` before the `String()` call.
- **Scope:** one-line
- **Status:** open

### BLK-3 — Non-string `$ref` crashes `_lookupByDotPath`

- **File:** `flint-mcp/src/core/dtcgTokenAdapter.ts:198-206, 284-287, 303`
- **Commandment:** 13
- **Observed:** Lines 198 and 284 check `'$ref' in rawValue` (key presence) but never `typeof rawValue.$ref === 'string'`. A payload `{ "$value": { "$ref": { "nested": "thing" } } }` passes the guard, then line 303 does `dotPath.split('.')` on a non-string, throwing `TypeError: dotPath.split is not a function`.
- **Rationale:** Same "adapter never throws" violation as BLK-2. Arrives via pure JSON (objects and arrays are JSON-native), so this one is reachable through any file-system input the adapter accepts — including Figma ingestion payloads and GPX pack imports. Lower exploit ceiling than BLK-1 (no DoS amplification), but still an unhandled exception in the MCP audit path.
- **Proposed fix:** Change the guard at lines 198 and 284 from `'$ref' in rawValue` to `'$ref' in rawValue && typeof (rawValue as {$ref: unknown}).$ref === 'string'`. Values that fail the guard should fall through to the "best-effort stringify" path (which BLK-2 is hardening).
- **Scope:** one-line
- **Status:** open

### WARN-1 — `_lookupByDotPath` traverses prototype chain via bracket access

- **File:** `flint-mcp/src/core/dtcgTokenAdapter.ts:299-310`
- **Commandment:** 13
- **Observed:** `current = (current as Record<string, unknown>)[part]` uses bracket access with no `Object.hasOwn` guard. A `$ref: "__proto__.toString"` walks into `Object.prototype.toString` (a function), then `isPlainObject(fn)` returns false at the next iteration → `undefined` → `ALIAS_BROKEN_REF`. **No write occurs.** But the read-path visits prototype nodes, and if *any other module on the same engine instance* has polluted `Object.prototype` with an own-named property (e.g. via a hostile npm dependency), the lookup would resolve to the polluted value.
- **Rationale:** Belt-and-suspenders hardening. Today's code path does not write to the prototype so pollution-to-RCE is not reachable through this adapter alone. But the invariant "token lookup only considers own properties of the document" is worth enforcing explicitly so that future refactors (e.g., returning the resolved leaf object for re-export) do not accidentally leak prototype keys.
- **Proposed fix:** Add `if (!Object.hasOwn(current as object, part)) return undefined;` before the bracket read. One-line change.
- **Scope:** one-line
- **Status:** open

### WARN-2 — O(N²) alias-chain visited-set via `Array.includes`

- **File:** `flint-mcp/src/core/dtcgTokenAdapter.ts:238, 242`
- **Commandment:** 13
- **Observed:** `visited: string[]` + `visited.includes(currentRef)` per hop. For an N-hop non-cyclic chain, total comparisons = N(N+1)/2.
- **Rationale:** At N=10,000 (plausible for a maliciously authored token file), this is ~50M string comparisons plus heap allocations per push. Synchronous CPU-blocking inside the MCP tool handler — stalls the Node event loop, blocks every other concurrent MCP request. Not a DoS primitive by itself but amplifies BLK-1's cost.
- **Proposed fix:** Replace with `const visited = new Set<string>([originPath])` and `visited.has(currentRef)` / `visited.add(currentRef)`. Also caps total hops via `if (visited.size > 1024) return { error: ALIAS_CYCLE... }` as a defense-in-depth guard.
- **Scope:** one-line
- **Status:** open

### SUG-1 — Hash-collision probability is not documented

- **File:** `flint-mcp/src/core/dtcgTokenAdapter.ts:63-75`
- **Commandment:** 13
- **Observed:** `fnv1a32` is 32-bit. The effective negative-id space after `| 0x80000000` is 2^31. For N tokens, collision probability by birthday approximation is ~N²/2^32. At N=1,000 it's ~0.0001%; at N=10,000 it's ~0.002%.
- **Rationale:** Probability is acceptable for realistic design systems (single-digit thousands of tokens), but the contract invariant `id-hash-collision-free` is aspirational, not proven. If two token_paths collide, the second-inserted token silently overwrites the first in any downstream `Map<id, DesignToken>` keyed on `DesignToken.id`. Currently no such map exists in the audit path, but this is a latent footgun.
- **Proposed fix:** Add a JSDoc block on `fnv1a32` documenting the collision probability and the invariant it assumes (downstream consumers key on `token_path`, not `id`). Alternately, escalate to a 64-bit BigInt hash — overkill for now. Document, don't over-engineer.
- **Scope:** one-line
- **Status:** open

## Verified Controls

- No `fs`, `child_process`, `path`, `os`, `crypto`, `http(s)`, or `net` imports — adapter is pure (Commandment 14 ✓)
- No regex on source code; regex not used anywhere in the adapter (Commandment 13 ✓)
- Hash ids are forced-negative via `| 0x80000000`; positive DB-issued ids are guaranteed disjoint
- Cycle-in-alias-chain terminates deterministically with typed error (happy path)
- Empty object, empty array, null, undefined, and non-object inputs return safely without throwing
- No MCP tool, no IPC channel, no preload surface added — attack surface against the process boundary is unchanged
- `server.ts:2038` integration is a single-line swap of a `.tokens` read — no new trust boundaries

## Recommendations (Priority Order)

1. **BLK-1** — Add depth guard to `_walkDTCG`; cap at 64
2. **BLK-3** — Tighten `$ref` guards to require `typeof === 'string'`
3. **BLK-2** — Wrap `String(rawValue)` fallback in try/catch or pre-guard Symbol/function types
4. **WARN-2** — Swap `Array<string>` visited-set for `Set<string>` and cap chain length at 1024
5. **WARN-1** — Add `Object.hasOwn` guard in `_lookupByDotPath`
6. **SUG-1** — JSDoc the collision-probability assumption on `fnv1a32`

All fixes are one-line to one-function scope. Estimated delta: ~20 lines across 5 functions. Re-review after patch should SHIP unless the fixes introduce new findings.
